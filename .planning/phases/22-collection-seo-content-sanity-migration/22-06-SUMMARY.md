# Phase 22 Plan 06: Markdown SEO importer Summary

Author a `.md`-source importer (`scripts/import-seo-md.mjs`) that mirrors the existing `.docx` importer but adds a `collectionSeo` upsert per article and a `--migrate-legacy` flag for the 6 existing post pairs — all verified via 11 passing vitest assertions.

## What Was Built

- **`scripts/import-seo-md.mjs`** (654 lines) — drop-in markdown counterpart to `scripts/import-seo-blog.mjs`. Reads 7 `.md` files from `/Users/mnmac/Downloads/` per an in-file `MANIFEST`, parses each with `marked` → JSDOM → `@portabletext/block-tools` (same pipeline as the docx flow), splits on the second `# H1`, and emits **4 mutations per file** inside one transaction: UA post, RU post, `translation.metadata`, and a `collectionSeo` mapping doc. Recognizes `--dry`, `--clean-old`, and `--migrate-legacy` CLI flags. `--migrate-legacy` upserts ONLY the 6 collectionSeo mappings — it never re-creates the legacy posts.
- **`tests/unit/import-seo-md.test.mjs`** — replaces the 5 `it.todo` placeholders from Plan 22-01 with 11 real assertions: MANIFEST length + handle inventory, LEGACY_MIGRATION shape, `buildCollectionSeo` field/ID conventions for both brand and gender surfaces, `processFile` rejection of files with ≠2 H1s, source-URL extraction, `buildPost` language and `hideFromBlogList` defaults, and proof that importing the module under vitest does **not** trigger CLI execution.
- **`package.json`** — `marked@^15.0.12` promoted from transitive dep to top-level `devDependencies`, no version bump (lockfile already pinned 15.0.12).

## Decisions Made

1. **Helpers copied verbatim from `import-seo-blog.mjs`, no shared module extracted.** CONTEXT D-07 explicitly says the two importers coexist and may diverge; refactoring to share helpers would couple the legacy importer to this plan's changes.
2. **Last LEGACY_MIGRATION entry uses `krosivky-ta-kedy` (not `cholovichi-krosivky-ta-kedy`).** CONTEXT D-14 is authoritative — the Shopify handle is shared between man and woman; gender disambiguation happens in the storefront URL, not the handle. RESEARCH Pattern 5 footnote confirms.
3. **`--clean-old` targets both `imported-seo-md-*` AND `imported-collection-seo-*` docs.** A single flag resets the importer's full footprint, which is what an operator running the importer iteratively expects.
4. **Handle verification against live Shopify deferred to Plan 07.** RESEARCH Open Q2 flagged that calling `getCollectionSlugs()` from a `.mjs` script would require either inlining a Storefront fetch or adding `tsx` as a dependency. The plan explicitly marks this as Plan 07's manual gate, so the script ships without it.
5. **Exports added at bottom of the script under the ESM CLI guard.** `if (import.meta.url === \`file://${process.argv[1]}\`)` runs `main()` only when invoked directly. Tests can `import('../../scripts/import-seo-md.mjs')` without triggering Sanity writes or CLI spam.
